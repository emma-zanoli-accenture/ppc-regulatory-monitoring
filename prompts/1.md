This is a SCRIPTED demo of a "Regulatory Change Management" tool for a Greek utility company. There is no backend — all data is static seed data in src/data/, strongly typed in src/types/. The narrative centers on ONE example regulatory change.

Domain context:
- A Compliance/Regulatory team monitors regulatory sources, interprets obligations, communicates required actions to impacted Business Units, and follows up until evidence is collected and the item is audit-ready.
- Two user roles use the tool: (1) Legal/Compliance User (analytical, control-tower view) and (2) Business Unit User (operational, action-oriented view).

Define TypeScript types and seed data for:

1. RegulatoryChange: id, title, regulatorySource, regulatoryArea, publicationDate, effectiveDate, implementationDueDate, riskPriority (Low/Medium/High), description, communicationStatus, implementationStatus, auditStatus.
2. Obligation: id, regChangeId, text, requirement, interpretation, scope, linkedProcess.
3. BusinessUnitImpact: id, regChangeId, businessUnit, impactedProcess, impactLevel, owner, dueDate, impactReason, selected (bool).
4. Ticket: id, regChangeId, businessUnit, status (use the workflow states below), viewed, downloaded, evidenceUploaded, completedOn, residualRisk, complianceMeasures (array), clarifications (array).
5. Evidence: id, ticketId, fileName, fileType, uploadedBy, uploadedOn, linkedObligationId.
6. AuditTrailEntry: id, regChangeId, ticketId?, timestamp, actor, action, detail.
7. SupportRequest / Clarification: id, ticketId, businessUnit, question, chatbotTranscript?, reportSection?, priority, status, legalResponse?, history.

Workflow states (single source of truth, export as a union type + display config with colors):
Draft Communication, Ready to Send, Notification Sent, Viewed, Downloaded, Clarification Requested, In Progress, Pending Evidence, Completed, Overdue, Escalated, Ready for Audit, Reported.

Seed ONE realistic example regulatory change for a Greek utility:
- Title: "REMIT II — Enhanced Wholesale Energy Market Transaction Reporting"
- Source: ACER / EU REMIT framework
- A realistic implementation due date a few months out (use a fixed date so the demo is deterministic; pick 30/09/2026).
- 3-4 obligations.
- Impacted Business Units (use these, NOT banking units): Energy Trading, Wholesale Market Operations, Procurement, Regulatory Affairs, Sustainability.
- Pre-built tickets in DIFFERENT states so dashboards look alive (some Sent, some Viewed, one Clarification Requested, one Completed/Ready for Audit, one approaching due date).
- A few pre-written audit trail entries.

Also add 2-3 additional regulatory changes as "background" rows (title + status only, no full detail) so list/dashboard views aren't empty.

Keep dates fixed and deterministic. Stop after the data layer is implemented and type-checks. Do not build UI yet.
